# AI 项目开发规范

本文档是本项目后续 AI 开发助手必须优先读取和遵守的项目规范。任何初始化、编码、重构、部署、依赖引入、数据库迁移、后台功能扩展，都应先对照本文档确认边界。

## 1. 项目定位

本项目是一个面向中国大陆访问者的内容与工具型网站，初期包含以下板块：

- 知识文档板块
- 在线实用工具板块
- AI 教程板块
- AI 提示词板块

项目后期会继续增加其他内容板块和功能，因此第一版必须保持清晰的工程边界、可扩展的数据模型、可维护的后台管理能力和规范的部署流程。

第一版目标不是堆砌复杂功能，而是建立一个稳定、可持续演进的基础平台：

- 公开前台用于内容展示、搜索、SEO 和在线工具访问。
- 后台控制台用于管理员管理内容、分类、标签、资源和发布状态。
- 后端 API 负责业务逻辑、权限校验、数据库访问和对象存储对接。
- PostgreSQL 保存结构化数据。
- 腾讯云 COS 保存图片、附件和其他资源文件。
- 腾讯云轻量应用服务器承载前端、后端、数据库和反向代理。

## 2. AI 执行原则

后续 AI 开发助手必须遵守以下原则：

- 开始任何开发前，先阅读本文档。
- 如果用户要求与本文档冲突，必须先指出冲突并请求确认。
- 不得在未确认的情况下更换主技术栈、数据库、部署方案或 UI 基础方案。
- 不得为了单个功能随意引入大型框架、外部服务或难以维护的依赖。
- 不得进行无明确目标的大规模重构。
- 每次修改前应先说明将影响的模块、文件范围和潜在风险。
- 优先遵循项目既有代码风格、目录结构、命名规则和抽象方式。
- 优先做小步、可验证、可回滚的变更。
- 涉及安全、权限、数据迁移、资源删除、部署配置时必须格外保守。
- 如果需求存在高影响歧义，应先向用户确认，而不是直接猜测。

## 3. 技术选型基线

除非用户明确批准变更，否则项目应采用以下技术基线：

- 前端与后端：Next.js 全栈项目
- 语言：TypeScript，开启 strict 模式
- UI：Tailwind CSS + shadcn/ui
- 数据库：PostgreSQL
- ORM 与迁移：Prisma
- 内容存储：数据库编辑与存储，不以 MDX 文件作为第一版主要内容源
- 对象存储：腾讯云 COS
- 部署编排：Docker Compose
- 反向代理与 HTTPS：Caddy
- 代码质量：ESLint、Prettier、Husky、lint-staged、提交信息规范
- 交付流程：Git + CI 自动部署为目标流程

技术选型原则：

- 以单机 4 核 4GB、40GB SSD、3M 峰值带宽的腾讯云轻量服务器为资源约束。
- 初期避免额外部署搜索服务、CMS 服务、消息队列、复杂监控平台等。
- 后期需要扩展时，先评估资源、维护成本和迁移成本。

## 4. 模块规划

项目应按以下逻辑模块规划，具体目录可在初始化项目时结合 Next.js 最佳实践确定：

- 公开前台：展示首页、栏目页、详情页、搜索页、工具页。
- 管理后台：管理员登录、内容管理、分类标签管理、资源库、发布控制。
- API 层：认证、权限、内容 CRUD、资源上传、搜索、后台统计。
- 数据库层：Prisma schema、迁移文件、种子数据、数据访问封装。
- 对象存储层：COS 上传、删除、访问 URL、资源元数据同步。
- 部署运维层：Docker Compose、Caddy、环境变量、备份、日志、健康检查。

模块边界要求：

- 前台页面不得直接绕过权限访问后台专用接口。
- 后台操作必须经过管理员认证和权限校验。
- 数据库访问应集中通过明确的数据访问层或服务函数完成。
- COS 文件操作必须和数据库资源记录保持一致。
- 工具类功能应尽量独立，避免和内容管理逻辑混杂。

## 5. 内容模型原则

知识文档、AI 教程、AI 提示词等内容第一版应采用统一内容模型，通过字段区分不同板块，而不是为每个板块重复建立完全独立的表和后台逻辑。

统一内容模型应至少支持以下概念：

- 板块类型，例如知识文档、AI 教程、AI 提示词。
- 标题、摘要、正文、封面图。
- slug 或其他稳定访问标识。
- 分类和标签。
- SEO 标题、SEO 描述、Open Graph 信息。
- 草稿、已发布、已下线等发布状态。
- 发布时间、创建时间、更新时间。
- 创建人、更新人等审计字段。
- 浏览量、排序权重等可选运营字段。

发布流程第一版采用：

- 草稿
- 预览
- 发布
- 下线

暂不实现复杂审核流。若后期出现多人协作、作者投稿、内容审核等需求，再设计作者、审核员、发布员等角色和状态机。

## 6. 后台管理原则

后台控制台第一版采用自研方案，不接入 Strapi、Directus 等独立 CMS。

后台第一版应优先包含：

- 管理员登录。
- 内容列表、创建、编辑、预览、发布、下线、删除或软删除。
- 分类和标签管理。
- COS 资源上传、浏览、复用、删除。
- 基础后台首页或统计概览。

后台安全要求：

- 第一版只开放管理员账号，不开放普通用户注册登录。
- 管理员密码必须使用可靠哈希算法存储，不得明文保存。
- 账号密码登录为第一版默认方案，但数据库和界面设计应预留 2FA 能力。
- 后台路由和 API 必须统一做认证校验。
- 敏感操作应保留审计字段，必要时记录操作日志。

## 7. 在线工具原则

在线实用工具板块第一版以前端浏览器本地运行为主，例如：

- JSON 格式化
- 编码转换
- 时间戳转换
- 文本处理
- 简单 AI 提示词辅助工具

工具开发原则：

- 能在浏览器安全完成的逻辑，优先不请求后端。
- 不上传用户输入内容，除非该工具明确需要后端处理且用户知情。
- 后端执行类工具必须先进行安全评估，尤其是代码执行、文件处理、网络请求、AI 调用等。
- 每个工具应具备清晰的标题、描述、输入、输出和错误提示。
- 后期如工具数量增加，可设计工具注册表或插件式工具结构。

## 8. 搜索与 SEO

第一版搜索使用 PostgreSQL 全文搜索，不额外部署 Meilisearch、Elasticsearch 等搜索服务。

搜索应优先覆盖：

- 标题
- 摘要
- 正文
- 标签
- 分类
- 板块类型

SEO 第一版应达到标准内容站要求：

- 服务端渲染或适合搜索引擎抓取的页面输出。
- 每篇内容可设置标题、描述、slug。
- 自动生成 sitemap。
- 提供 robots.txt。
- 支持 Open Graph 基础信息。
- 重要内容页具备清晰语义结构。
- 面向中国大陆访问，后续应考虑 ICP 备案、中文搜索引擎收录、COS/CDN 访问策略。

高级 SEO 能力，如自动内链、专题页、结构化内容评分、关键词策略等，留待内容规模扩大后再设计。

## 9. 资源与对象存储

所有用户上传或后台上传的资源文件必须存储在腾讯云 COS，不得长期存放在应用服务器目录。

资源库应在数据库保存元数据，包括：

- 文件名
- 文件类型
- MIME 类型
- 文件大小
- COS key
- 访问 URL
- 上传人
- 创建时间
- 关联内容信息

资源处理要求：

- 上传前校验文件大小和类型。
- 图片、附件等类型应有明确白名单。
- 删除资源前确认是否被内容引用。
- 不得在代码中硬编码 COS 密钥、bucket、region 等敏感配置。
- COS 访问策略应优先使用最小权限。

## 10. 安全规范

必须遵守以下安全要求：

- 所有密钥、数据库连接串、COS 凭证、JWT 或 session secret 都必须通过环境变量配置。
- 不得提交 `.env`、密钥、证书、数据库备份等敏感文件。
- 所有后台 API 必须校验管理员身份。
- 所有数据库写入必须经过服务端校验。
- 防止 SQL 注入，优先使用 Prisma 安全查询方式。
- 防止 XSS，富文本或 Markdown 渲染必须经过安全处理。
- 防止 CSRF，后台写操作应采用合适的 CSRF 或同等防护策略。
- 上传接口必须限制文件大小、类型和访问权限。
- 登录接口应考虑速率限制和错误信息收敛。
- 生产环境必须开启 HTTPS。

## 11. 部署规范

目标部署环境：

- 腾讯云轻量应用服务器
- 4 核 CPU
- 4GB 内存
- 40GB SSD
- 3M 峰值带宽
- 腾讯云 COS 对象存储

第一版部署方式：

- 使用 Docker Compose 编排服务。
- Next.js 应用、PostgreSQL、Caddy 部署在同一台服务器。
- Caddy 负责反向代理和 HTTPS。
- PostgreSQL 数据必须使用持久化 volume。
- 上传资源不进入服务器本地持久目录，统一进入 COS。

部署规范：

- 生产环境使用独立环境变量文件或服务器密钥管理方式。
- 数据库迁移必须通过 Prisma migration 执行，不得手动随意改表。
- 每次发布前应完成类型检查、lint、构建和必要测试。
- 发布失败应能回滚到上一版本镜像或上一稳定提交。
- Caddy、应用、数据库日志应能被查看和定位。

## 12. 备份与恢复

PostgreSQL 必须设计每日自动备份，备份文件上传到腾讯云 COS。

备份规范：

- 每日至少备份一次生产数据库。
- 备份文件应包含日期时间标识。
- 备份应设置保留周期，初期可按 7 天、30 天、90 天分层保留。
- 定期验证备份文件可恢复。
- 恢复流程必须写入部署文档或运维文档。
- 数据库备份文件不得提交到 Git 仓库。

COS 中的资源文件也应关注误删风险。后期可根据成本和重要性开启版本控制、生命周期规则或跨区域备份。

## 13. 监控与日志

第一版采用基础自托管监控，不强制接入外部 SaaS。

至少应具备：

- Docker 容器健康检查。
- 应用结构化日志。
- 后台错误日志。
- Caddy 访问日志和错误日志。
- PostgreSQL 基础运行日志。
- 磁盘空间告警。
- 内存和 CPU 使用情况观察。

如后期线上问题定位成本升高，可再评估接入 Sentry、腾讯云日志服务或其他监控平台。

## 14. 开发规范

代码质量要求：

- TypeScript 必须开启 strict 模式。
- 提交前运行 lint、format、typecheck。
- 使用 Prettier 保持格式一致。
- 使用 ESLint 约束代码质量。
- 使用 Husky 和 lint-staged 做提交前检查。
- 使用规范化提交信息，便于追踪变更。

数据库开发要求：

- 所有表结构变化必须通过 Prisma migration。
- migration 文件应随代码提交。
- 不得在生产数据库直接手工改结构。
- 重要迁移前必须考虑数据备份和回滚方案。

测试要求：

- 关键业务逻辑应有单元测试或集成测试。
- 后台权限、内容发布、资源上传、数据库迁移应重点测试。
- 发布前至少完成类型检查、lint 和构建验证。

协作要求：

- 不在一个提交中混杂大量无关修改。
- 不随意格式化无关文件。
- 不删除用户已有代码或配置，除非明确确认。
- 变更公共接口、数据库结构、部署配置前必须说明影响。

## 15. 后续抉择点清单

以下事项尚未在第一版实现前完全确定，后续遇到时必须再次询问用户或形成新的技术方案：

- 是否开放普通用户注册登录。
- 是否支持收藏、评论、点赞、浏览历史。
- 是否引入内容审核流。
- 是否支持多管理员角色和更细权限。
- 是否将搜索升级为 Meilisearch、Elasticsearch 或其他搜索服务。
- 是否接入 AI API，用于内容生成、提示词优化、摘要、翻译等。
- 是否增加会员、支付、积分或付费内容。
- 是否支持多语言站点。
- 是否接入 CDN。
- 是否将 PostgreSQL 迁移到云数据库。
- 是否将文件访问改为私有读加临时签名 URL。
- 是否接入 Sentry、腾讯云日志服务或其他 SaaS 监控。
- 是否加入专题页、自动内链、内容推荐等高级 SEO 能力。
- 是否引入独立 CMS 或低代码后台。
- 是否将在线工具扩展为插件式架构。
- 是否需要移动端 App、小程序或开放 API。

## 16. 禁止事项

除非用户明确批准，禁止执行以下行为：

- 不要初始化项目结构，除非用户明确要求开始初始化。
- 不要创建 `package.json`，除非用户明确要求初始化项目。
- 不要安装依赖，除非用户明确要求。
- 不要把上传资源长期存储在应用服务器目录。
- 不要跳过 Prisma migration 直接修改数据库结构。
- 不要在代码中硬编码密钥、数据库连接串、COS 凭证。
- 不要提交 `.env`、证书、备份文件、私钥等敏感文件。
- 不要未经确认改变 Next.js、PostgreSQL、Prisma、Tailwind、shadcn/ui、Docker Compose、Caddy 等核心选型。
- 不要为了小功能引入大型复杂服务。
- 不要无计划地重构目录结构或公共接口。
- 不要在没有备份方案的情况下执行高风险数据操作。
- 不要在没有安全评估的情况下添加后端执行类在线工具。

第一版默认包含：

- 公开内容展示。
- 统一内容模型。
- 管理员后台。
- 内容草稿、预览、发布、下线。

后续 AI 开发助手应以本文档作为项目长期演进的基线，并在每次新增功能前确认该功能是否仍符合第一版边界或需要进入新的规划阶段。

## 21. 本地开发与 Docker 部署流程规范

本地 Docker Desktop 和 Docker Compose 的作用：

- Docker Desktop 提供本地 Docker Engine、镜像管理、容器运行环境和图形化查看能力。
- Docker Compose 用于在本地一次性启动多服务环境，例如 Next.js 应用、PostgreSQL、Caddy 或其他依赖服务。
- 本地开发时可以用 Docker Compose 模拟生产环境，避免每台电脑手动安装 PostgreSQL、Caddy 等运行依赖。
- 本地开发环境默认不在宿主机裸装 PostgreSQL；需要数据库时，通过 Docker Compose 启动 PostgreSQL 容器。
- 本地 Docker 主要用于开发、调试、构建验证和部署前预演；生产运行仍以云服务器 Docker Compose 为准。

必须区分以下概念：

- Git 仓库保存源代码、配置模板、Dockerfile、Compose 文件和迁移文件。
- Docker image 是根据 Dockerfile 构建出来的可运行包。
- Docker container 是 image 启动后的运行实例。
- 服务器部署时通常拉取或构建 image，再通过 Docker Compose 启动 container；不要把 container 当作代码仓库或文件同步方式。

推荐开发流程：

1. 本地创建 Git 仓库并初始化项目代码。
2. 本地使用 Next.js 开发前台、后台和 API。
3. 本地使用 Docker Compose 启动 PostgreSQL 等依赖服务。
4. 本地运行类型检查、lint、测试、数据库迁移和构建验证。
5. 代码提交到 Git 仓库。
6. CI 根据 Git 提交构建 Docker image。
7. CI 将 image 推送到镜像仓库。
8. 云服务器通过 SSH 或部署脚本执行 `docker compose pull` 和 `docker compose up -d`。
9. 服务器只运行容器和持久化数据，不直接作为主要开发环境。

第一阶段允许采用简化部署流程：

- 如果暂未配置 CI 和镜像仓库，可以先由服务器从 Git 仓库拉取代码，在服务器上执行 `docker compose build` 和 `docker compose up -d`。
- 简化流程只适合项目早期体验和低频发布。
- 一旦进入稳定开发，应迁移到“CI 构建 image、服务器 pull image”的流程。

本地运行方式建议：

- 开发体验优先时，可以本地 Node.js 运行 `next dev`，同时用 Docker Compose 运行 PostgreSQL。
- 环境一致性优先时，可以用 Docker Compose 同时运行 Next.js 和 PostgreSQL。
- 无论选择哪种方式，生产部署必须通过 Docker Compose 管理。

部署注意事项：

- PostgreSQL 生产数据必须使用 Docker volume 或明确的持久化目录。
- COS 资源不得进入应用容器或服务器本地长期存储。
- `.env`、生产密钥、数据库密码、COS 密钥不得提交到 Git。
- 每次新增服务或端口前，必须检查 Docker Compose、UFW 和腾讯云安全组。
- 正式部署 Caddy 或项目应用前，必须先停止并移除临时体验容器 `html-demo`。

## 22. UI 风格与网站布局设计规范

本项目后续公开前台、后台控制台、内容页面、工具页面和运营页面的 UI 风格，统一参考 StyleKit 的 Corporate Clean 模板。

参考来源：

- 模板页面：`https://www.stylekit.top/zh/styles/corporate-clean/showcase`
- 本地硬性提示词文件：`corporate-clean-hard-prompt.md`

执行要求：

- 后续进行任何 UI 设计、页面开发、组件开发或视觉调整前，必须先读取 `corporate-clean-hard-prompt.md`。
- 项目整体采用 Corporate Clean 企业简洁风，强调专业、可信、克制、清晰、可读、功能优先。
- UI 风格必须服务于内容站、工具站和后台管理系统的长期使用效率，避免营销页式过度装饰。
- 同一页面内颜色、间距、圆角、阴影、边框、按钮、表单和卡片必须保持一致。
- 前台和后台可以有信息密度差异，但必须共享同一套基础视觉语言。

Corporate Clean 核心风格：

- 使用干净的白色和浅灰背景，页面背景优先使用 `slate-50` 或接近色。
- 主色使用企业蓝，主要 CTA 优先使用 `blue-600`，hover 使用更深蓝。
- 正文和标题使用 slate 系中性色，标题优先深色，辅助文本使用较浅灰蓝。
- 使用充足留白、清晰网格和稳定的 8px 间距体系。
- 卡片和容器使用轻圆角，例如 `rounded-lg`、`rounded-xl`。
- 阴影保持克制，优先 `shadow-sm`，悬停最多使用 `shadow-md`。
- 交互动效保持快速轻量，常规过渡时间控制在 150ms 到 200ms。
- 所有交互元素必须有清晰的 hover、active、focus 状态。
- 表单聚焦态优先使用 `focus:ring-2 focus:ring-blue-500` 或同等蓝色聚焦样式。
- 布局优先清晰的信息层级，而不是装饰性视觉冲击。

网站布局方向：

- 首页第一屏应直接传达站点定位和核心入口，不做空泛营销式 hero。
- 栏目入口应清晰展示知识文档、在线工具、AI 教程、AI 提示词等核心板块。
- 内容页面优先保证阅读体验，正文区域宽度、字号、行高和目录导航要稳定。
- 工具页面优先保证操作效率，输入区、输出区、操作按钮、错误提示要清楚。
- 后台管理端应更高信息密度、更少装饰，优先表格、筛选、表单、状态标签和批量操作。
- 不使用页面大面积渐变、装饰光斑、漂浮图形、强烈玻璃拟态或复杂插画作为主要风格。

禁止风格漂移：

- 禁止使用霓虹色、赛博风、厚重渐变、过度阴影、强装饰背景。
- 禁止把后台管理端做成营销落地页风格。
- 禁止使用 `shadow-2xl`、复杂自定义 box-shadow 或强烈立体阴影作为常规组件样式。
- 禁止多个高饱和强调色在同一视图中竞争视觉焦点。
- 禁止正文或常规 UI 文本使用 `font-mono`，除非是代码、命令、日志、密钥片段等技术内容。
- 禁止使用黑色粗边框、荧光粉色硬阴影、强烈错位阴影等与 Corporate Clean 页面展示不一致的风格作为项目默认 UI。

冲突处理：

- 如果 `corporate-clean-hard-prompt.md` 内部规则互相冲突，优先遵循模板页面展示的 Corporate Clean 企业简洁风。
- 如果本地硬性提示词中的片段与“蓝色主色、浅色背景、轻圆角、轻阴影、8px 间距、无装饰”的方向冲突，必须先向用户确认再采用。
- 如果用户后续明确更换 UI 风格，必须先更新本文档和对应硬性提示词文件，再进行页面实现。

## 24. 产品开发路线与首页规划规范

本项目虽然包含多个板块，但开发节奏必须保持单点聚焦。后续开发时一次只重点推进一个板块，不同时铺开多个复杂模块，避免早期架构和内容维护失控。

核心板块方向：

- 文档：包含 AI 教程、AI 提示词、AI Skills 等知识内容。
- 在线实用工具：偏通用工具，例如格式化、转换、检测、计算类工具。
- 一站式导航页：聚合高价值站点、资源、工具和专题入口。
- AI 工具：面向明确结果交付的 AI 生成工具，是后续主要盈利方向。

开发优先级：

- 当前主要开发目标是首页。
- 首页完成后，再围绕一个具体板块做完整闭环。
- 不要在首页阶段同时实现完整文档系统、完整导航系统、完整 AI 工具商业化和完整用户付费系统。
- 每个板块进入开发前，应先明确该板块的 MVP 范围、数据模型、后台管理需求、前台展示形态和验收标准。

AI 工具产品原则：

- AI 工具应专注于给用户生成具体解决方案，而不是做成通用 AI 聊天窗口。
- 工具形态应是“用户填写参数 -> 系统生成结果 -> 用户查看、复制、下载或继续优化”。
- 示例方向包括 AI 生成爆款文案、AI 生成视频脚本、AI 生成电商主图等。
- 每个 AI 工具必须有明确的输入参数，例如商品名称、卖点、平台、风格、尺寸、数量、语气、目标人群等。
- AI 工具可接入中转 API，但不得在前端暴露 API 密钥。
- AI 工具生成流程应对用户隐藏复杂提示词和模型细节，前台只暴露必要参数和结果。
- AI 工具页面应强调结果质量、使用效率和商业价值，不强调聊天陪伴感。

AI 工具盈利方向：

- 后续主要盈利方式来自 AI 工具。
- 可采用免费次数加积分付费模式。
- 可采用免费生成一版，更多版本或高清结果付费查看模式。
- 可采用基础结果免费，高级参数、批量生成、导出、历史保存等能力付费模式。
- 付费、积分、订单、权益和风控系统不在首页阶段实现，进入 AI 工具商业化阶段前必须单独规划。

首页定位：

- 首页是整个网站的整合式入口页，不是单纯营销落地页。
- 首页应帮助用户快速进入文档、工具、导航、AI 工具等核心区域。
- 首页应兼顾信息发现、搜索、核心板块入口和热门 AI 工具转化。
- 首页第一版优先做静态或半静态内容结构，后台动态管理能力可在后续板块开发时逐步补齐。

首页推荐布局：

- 顶部导航：左侧 logo，右侧展示核心板块目录，例如 AI生成、AI文档、在线工具、一站式导航。
- 首屏中心：放置主搜索框，支持站内、Bing、Google 等搜索切换，当前首页默认搜索引擎为 Bing。
- 搜索框下方只展示四个核心入口：AI生成、AI文档、在线工具、一站式导航。
- 主内容区优先顺序为：AI生成整宽重点展示区、最新文档资讯流与一站式导航并列、在线工具整宽区域。
- AI生成必须独占整行宽度，但首页第一版不使用轮播、左右箭头或底部脚标；首屏只展示 3 个核心卡片，右上角使用“更多”文本入口，为后续工具列表页或展开页预留空间。
- AI生成区不使用外围边框容器，避免边框过多造成视觉臃肿；只保留内部工具卡片的轻量边框和层级。
- 最新文档应使用紧凑资讯流，不使用连续大卡片，避免首页看起来像博客列表。
- 首页最新文档列表只展示文档标签和文档主标题，不展示摘要、副标题或日期；标题默认使用浅色文字，悬浮时使用主题色高亮和动效下划线。
- 一站式导航用于跳转其他网站和外部资源，不是本站内部导航模块；第一版可用知名网站作为模板数据。
- 在线工具位于底部整宽区域，作为后续工具集合入口；首页在线工具区不使用外围边框容器，只保留内部工具卡片层级。
- 移动端应改为单列排序，优先顺序为搜索、核心入口、AI生成、最新文档、一站式导航、在线工具。

首页设计注意事项：

- 不做夸张 hero，不使用大面积口号式营销视觉。
- 搜索是首页核心交互之一，应足够醒目但保持 Corporate Clean 风格。
- AI生成入口应比普通文档和导航入口更突出，但不能破坏整体克制企业风。
- AI生成卡片内的操作按钮文本使用“去使用”，按钮默认使用低强调浅色样式并靠右排列，悬浮时再显示主题色高亮；按钮和文档标题等可点击元素悬浮时必须使用 `cursor: pointer`。
- 首页标题允许使用克制的彩色点缀元素降低单调感；大标题和分区标题的横线点缀应放在标题文字底部背后，横线底部与标题底部对齐，形成类似涂鸦/荧光笔划过的效果；不要在标题旁额外添加圆点或小色块点缀。AI生成卡片标题的左侧细竖条可以保留。
- AI生成主标题前使用项目已安装的 `lucide-react` 星星类图标，例如 `Sparkles`，呈现类似 Gemini 的 AI 感；不要为单个标题装饰引入需要额外授权或署名的外部图标素材。
- 文档资讯流用于建立内容可信度，AI生成卡片用于后续商业转化。
- 首页第一版可以先使用占位数据或静态配置，但必须为后续数据库驱动预留结构。

后续抉择点：

- 站内搜索第一版是仅搜本地内容，还是同时支持外部搜索跳转。
- AI 工具是否需要登录后使用。
- 免费次数按游客、账号、IP、设备还是组合策略限制。
- 积分系统、支付系统和订单系统采用何种实现。
- AI 中转 API 的服务商、计费方式、限流策略和失败兜底。
- AI 工具生成结果是否保存历史记录。
- AI 工具是否支持公开案例、模板市场或用户作品库。

- 首页 AI生成卡片的操作按钮文本为“去使用”，按钮默认保持低强调浅色并靠右排列，悬浮时显示主题色高亮，并使用 pointer 鼠标指针。
- 在线工具入口仅作为第一批工具规划展示，不实现具体工具逻辑。

首页设计约束：

- 首页继续遵守 Corporate Clean 企业简洁风。
- 首页不做夸张营销 hero，不使用大面积渐变、装饰光斑或强烈阴影。
- 首页第一屏不展示大标题介绍语、阶段说明或路线图卡片，直接以搜索入口和快捷入口作为主要交互。
- 首页搜索区不使用外围卡片边框；搜索引擎切换以文本形式排列在搜索框上方，例如“站内、bing、Google”，不显示图标。
- 搜索引擎当前状态通过咬合搜索框上边框的主题色三角箭头表达，箭头必须使用主题蓝色，不使用灰色。
- 搜索引擎指示箭头移动过渡使用 GSAP；首次进入页面时箭头直接定位到默认搜索引擎下方，不播放从左侧滑入，后续切换搜索引擎时保留滑动动效。
- 搜索引擎指示箭头在首次完成定位前必须隐藏，避免页面加载瞬间出现在搜索框最左侧。
- 搜索按钮内嵌在输入框右侧。
- 站内搜索的“站内推荐”或搜索结果必须使用悬浮浮层展示，不占用正常页面结构高度，不得把搜索框下方的核心入口挤开；浮层所在搜索区的层级必须高于后续首页内容区，避免被 AI生成等板块遮挡，同时顶部导航仍应保持更高层级。
- 站内推荐列表项保持稳定直接显示，不使用错峰滑入等入场动效；搜索浮层这类核心反馈区域优先保证可见性和可靠性。
- Bing 和 Google 搜索模式下支持搜索联想词，联想列表必须使用悬浮浮层展示，不占用正常页面结构高度。
- 搜索联想通过本项目 Next API 路由代理外部搜索引擎联想接口，前端不得直接请求第三方联想接口，避免浏览器跨域和接口变更带来的页面错误。
- 搜索联想输入处理使用防抖，当前默认约 `260ms`；不要使用会持续打点请求的节流方案。中文拼音输入法组合输入期间不得触发联想请求，必须等待 `compositionend` 后再按最终上屏文本请求。
- 搜索联想接口必须有超时、数量限制、去重和失败空列表兜底；外部搜索引擎不可用时不得影响搜索框正常提交。Google 联想域名在中国大陆或部分 Node 运行环境中可能不可达，此时应返回空列表，不阻塞 Google 搜索跳转。
- 首页所有卡片、按钮、输入框必须保持轻圆角、轻阴影、蓝色聚焦态和清晰层级。
- 首页最新文档资讯流只展示文档标签和文档主标题，不显示摘要、副标题或日期；标题默认使用浅色文字，悬浮时使用主题色高亮和动效下划线，并使用 pointer 鼠标指针。
- 首页各级标题应使用克制的彩色点缀元素区分层级和板块，优先使用标题底部涂鸦线、图标徽标、卡片标题细竖条等轻量元素；不使用标题旁圆点、小色块、大面积渐变或装饰光斑。
- 首页装饰性图标优先使用项目已安装图标库，避免从 Icons8 等需要署名或付费授权的网站引入外部图标；如确需外部素材，必须先确认授权方式、署名要求和商业使用范围。
- 首页可使用轻量背景动效降低单调感，但只能使用低透明度细线、轻网格、几何轮廓等克制元素；不得使用装饰光斑、渐变球、粒子爆炸或强视觉动态。
- 首页背景动效必须位于内容底层，不影响阅读、点击、搜索、卡片层级和页面性能。
- 背景动效必须支持 `prefers-reduced-motion: reduce`，用户偏好减少动画时停止漂移动效。
- 后续接入真实数据时，应优先替换数据来源，不重写首页整体结构。

本地开发注意事项：

- 如果浏览器控制台持续输出 Next.js HMR WebSocket 连接失败，必须先查看 dev server stderr 日志；Next.js 16 会默认阻止未加入白名单的开发源访问 `/_next/webpack-hmr`。
- 本项目本地开发常用访问地址为 `http://127.0.0.1:3000`，因此 `next.config.ts` 必须保留 `allowedDevOrigins: ["127.0.0.1"]`，避免 HMR 被 Next dev origin 防护拦截。
- 修改 `next.config.ts` 后必须重启本地 dev server，刷新页面后再判断 HMR 是否恢复。
- 需要无 HMR 干扰的纯预览时，可使用 `npm run build` 后运行生产预览服务。
